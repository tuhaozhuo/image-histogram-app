# 图像灰度直方图 · 准确性测试报告

| 项 | 内容 |
|---|---|
| 测试日期 | 2026-07-08 |
| 被测对象 | `native/histogram.cpp` 计算核心（v1.2.0） |
| 测试环境 | macOS arm64（Apple Silicon），clang，`-O3 -ffp-contract=off` |
| 测试目的 | 验证灰度化 + 统计 + 归一化结果与标准公式 `round(R×0.299+G×0.587+B×0.114)` 一致 |

> **平台适用性说明**：本测试在 macOS arm64 上进行。因 Android/iOS 双端复用**同一份** `histogram.cpp`，并统一用 `-ffp-contract=off`（禁 FMA 融合）保证浮点结果跨平台确定一致，故本报告结论同样适用于双端真机。穷举验证（测试 B）验证的是与平台无关的算法逻辑。

## 1. 测试方法

采用三层交叉验证，从完备到抽样：

| 层 | 方法 | 覆盖 |
|---|---|---|
| A | 已知输入用例 | 手算期望值可对照的构造用例（灰度公式/归一化/alpha/边界） |
| B | **灰度化穷举完备验证** | 全部 256³ = 16,777,216 种 (R,G,B)，非抽样 |
| C | 整图三实现对拍 | 合成图，优化实现 vs 独立标准公式参考实现，逐 bin |
| D | 真实图片对拍 | 真实照片，逐 bin |

判定标准：默认实现（多线程标量）与标准公式**逐 bin 完全一致（bin-exact）**。

## 2. 测试结果

### 测试 A：已知输入准确性用例 —— 全部 PASS ✓

**灰度公式各通道**（`gray = round(R×0.299+G×0.587+B×0.114)`）：

| 用例 | 输入 RGB | 期望 gray | 实测 | 结果 |
|---|---|---|---|---|
| 纯黑 | (0,0,0) | 0 | bin0=100 | ✓ |
| 纯白 | (255,255,255) | 255 | bin255=100 | ✓ |
| 纯红 | (255,0,0) | 76（255×0.299=76.245） | bin76=100 | ✓ |
| 纯绿 | (0,255,0) | 150（255×0.587=149.685） | bin150=100 | ✓ |
| 纯蓝 | (0,0,255) | 29（255×0.114=29.07） | bin29=100 | ✓ |
| 中灰 | (128,128,128) | 128 | bin128=100 | ✓ |

**归一化**（`out[i] = round(count[i]/max×100)`）：

| 用例 | 构造 | 期望 | 实测 | 结果 |
|---|---|---|---|---|
| 黑白各半 | hist[0]=hist[255]=n/2 | bin0=100 & bin255=100 | 100, 100 | ✓ |
| 比例 1000:250 | gray=100 出现 1000 次、gray=50 出现 250 次 | bin100=100 & bin50=25 | 100, 25 | ✓ |

**边界**：

| 用例 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 空图（max==0） | 全 0、不除零、不崩溃 | 全 0 | ✓ |

### 测试 B：灰度化穷举完备验证 —— PASS ✓

枚举全部 **16,777,216** 种 (R,G,B) 组合，对比各实现与标准公式：

| 实现 | 与标准公式不一致数 | 占比 | 结论 |
|---|---|---|---|
| **标量 (double)** | **0** | 0% | **完全一致 ✓ (bin-exact)** |
| NEON (float32) | 3,712 | 0.0221% | 最大偏差 1 灰度级 |

> 这是**完备证明**（全覆盖，非抽样）：默认实现的灰度化对**任意可能的像素**都与标准公式完全相同。NEON 的微小偏差源于 float32 精度低于 double，仅在灰度真值卡在半整数边界的极少数像素上四舍五入分歧，属性能/精度权衡。

### 测试 C：整图三实现对拍 + 性能 —— PASS ✓

| 图像 | 实现 | 耗时 | 加速比 | 精度（vs 标准公式） |
|---|---|---|---|---|
| 合成 12MP | 单线程标量 | 9.96 ms | 1.00× | **bin-exact ✓** |
| | 多线程标量 | 2.45 ms | 4.06× | **bin-exact ✓** |
| | NEON 多线程 | 1.88 ms | 5.30× | 5 bins ±1 |
| 合成 24MP | 单线程标量 | 20.06 ms | 1.00× | **bin-exact ✓** |
| | 多线程标量 | 4.81 ms | 4.17× | **bin-exact ✓** |
| | NEON 多线程 | 3.65 ms | 5.49× | 3 bins ±1 |

边界用例：纯黑图 `out[0]==100 & 其余 0` → PASS；空图 `max==0` 防除零 → PASS。

### 测试 D：真实图片对拍 —— PASS ✓

| 图像 | 实现 | 耗时 | 精度 |
|---|---|---|---|
| 荷花样张 1.9MP | 单线程/多线程/NEON | 2.03 / 0.55 / 0.43 ms | **三者均 bin-exact ✓** |
| 系统壁纸 Sonoma 4.0MP | 单线程/多线程/NEON | 7.04 / 1.52 / 1.48 ms | **三者均 bin-exact ✓** |

> **观察**：真实照片上连 NEON 也 bin-exact（0 偏差）。因为真实照片的像素灰度极少恰好卡在半整数边界，NEON 的 float32 舍入分歧几乎不发生——NEON 的 ±1 只在渐变类规律合成图上才明显。**真实使用场景下三种实现精度实际等价。**

## 3. 透明图 alpha 处理（专项）

计算核心**忽略 alpha 通道**，只用 RGB 算灰度（测试 A 已验证：半透红 a=128、全透红 a=0 与不透红同为 bin76）。

但需注意一个**端到端口径**：Flutter 的 `dart:ui` 解码半透明图时会**预乘 alpha**，导致到达计算核心的 RGB 已被改变。实测：

| 图像 | 原始 RGBA | Flutter 解码后 | 灰度 |
|---|---|---|---|
| 半透明红 | (255,0,0,α=128) | 预乘 → (128,0,0) | 38 |
| 不透明红 | (255,0,0,255) | (255,0,0) | 76 |

因此同为"红色"的两张图，在 app 内直方图柱位置不同（bin38 vs bin76）。**这不是计算错误**（核心对相同输入必然相同输出），而是"透明图 + 解码器预乘策略"的组合效应。cv2 / stb_image 不预乘（结果为 76），Flutter 预乘（结果为 38）。

**口径建议**：若需与 OpenCV/标准工具的"原始色"口径一致，在 Dart 端解码后还原预乘（un-premultiply）；否则在使用说明中标注"透明图按解码后（合成显示色）统计"。

---

## 4. 结论

1. **默认实现（多线程标量 double）的灰度化对全部 1677 万种像素与标准公式完全一致（bin-exact），准确性达标。**
2. 统计、归一化、边界处理均通过已知用例验证，结果符合手算期望。
3. NEON 实现有 0.0221% 的 ±1 微小偏差（float32 精度权衡），仅用于性能对比展示；真实图片上实测 0 偏差。默认主流程不使用 NEON，故不影响准确性。
4. 透明图的表现取决于上游解码器的 alpha 预乘策略，属已知口径，非计算错误。

**准确性验证：通过 ✓**
