## 1. 项目是什么

课题「图像直方图计算及性能优化」。业务流程：**用户上传图片 → 高性能计算灰度直方图（≤300ms）→ 显示 256×100 黑白直方图 + 生成耗时**。

评分只看两点：
- **速度**：直方图生成耗时 ≤ 300ms
- **准确性**：灰度化与统计结果与标准公式一致

一句话架构：**一套 Flutter UI（Android/iOS 双端）+ 一份共享 C++ 计算核心（dart:ffi 调用）**。计算核心是第一等公民，双端复用同一份 `native/histogram.cpp`。

## 3. 核心算法讲解

直方图计算分四步（对应 `hist_compute_rgba` 内部）：

### 步骤 1：灰度化（统计时灰度化）

对每个像素 `(R, G, B)`，用**标准公式**算出一个灰度值：

```
gray = round(R × 0.299 + G × 0.587 + B × 0.114)
```

- `0.299 / 0.587 / 0.114` 是 BT.601 亮度权重（人眼对绿最敏感，故 G 权重最大）。三者之和 = 1.0，所以 gray 落在 0–255。
- `round` 是四舍五入取整。
- 「统计时灰度化」= 一边遍历一边算灰度并计数，**不生成中间灰度图**，省一次内存读写。

### 步骤 2：统计（累加到 256 个桶）

维护一个长度 256 的整数数组 `hist`。每算出一个 gray，就 `hist[gray]++`。遍历完，`hist[i]` = 灰度值为 `i` 的像素个数。

### 步骤 3：归一化到 0–100

找出 `hist` 中的最大值 `max`，把每个桶按比例映射到 0–100：

```
out[i] = round(hist[i] / max × 100)
```

出现次数最多的那个灰度级归一化后 = 100，其余按比例缩小。**特判 `max == 0`**（空图）→ 全 0，避免除零。

### 步骤 4：绘制

把 256 个归一化值（0–100）画成 256 根竖柱，柱高 = 归一化值。这就是「256×100 黑白直方图」。

### 计时口径（重要）

耗时**只计步骤 2 + 3**（统计 + 归一化），**不含**图像解码、内存拷贝、UI 绘制。核心函数内部用 `std::chrono::steady_clock` 只包住这两步，返回毫秒。这样测的是纯算法性能，对齐评分口径。

## 4. 三种实现的原理与精度

app 内提供三种实现做性能对比。**先说结论纠正常见误解：不是三种都有精度损失——单线程标量和多线程标量都是精确的（bin-exact），只有 NEON 有 ±1 的微小损失。**

### 方法一：单线程标量（baseline）

**原理**：一个线程，从头到尾逐像素遍历。每个像素用 `double` 算 `gray = R*0.299+G*0.587+B*0.114`，`std::lround` 取整，`hist[gray]++`。

最直接、最慢，作为性能基准（加速比的分母）。

**精度**：用 double（双精度）严格算标准公式 → **精确**。

### 方法二：多线程标量

**原理**：把图像**按行切成 N 块**（N = CPU 核数），每个线程处理一块。关键：**每个线程维护自己独立的一份 256 直方图（局部直方图）**，各写各的，算完后主线程把所有局部直方图**逐桶整数相加**归并成全局直方图。

为什么要局部直方图？如果多线程共享同一个 `hist` 数组做 `hist[gray]++`，会发生**数据竞争**（两个线程同时改同一个桶），必须加锁或用原子操作，反而拖慢。局部直方图各自独立 → **无锁**，最后归并一次即可。

**精度**：每个线程内部还是 double 标准公式，归并是整数加法（完全精确）→ 结果与单线程**逐桶完全相同**，**精确**。多线程只影响速度，不影响结果。

**加速比**：理论上接近核数，实际约 3–4×（受内存带宽限制，因为直方图统计是内存密集型）。

### 方法三：NEON 多线程

**原理**：在多线程基础上，把**灰度化**这一步用 NEON（ARM 的 SIMD 指令集）向量化。SIMD = 单指令多数据，一条指令同时算多个像素。

具体做法（见 `accumulate_neon`）：
1. `vld4_u8` 一次加载 8 个像素的 RGBA，自动分离成 R/G/B/A 四个向量。
2. 把 R/G/B 转成 `float32` 向量，用向量乘加一次算出多个像素的灰度（`float32x4` 一次 4 个，8 像素分两批）。
3. round 转成整数，存到临时数组。
4. **再用标量循环**把这 8 个 gray 值累加到直方图 `hist[gray]++`。

**为什么累加还是标量？** 直方图累加是 **scatter（散射写入）**——每个像素的 gray 不同，要写入直方图数组的不同位置。SIMD 擅长「规则的、连续的」运算（如灰度化的乘加），但不擅长「每个元素写到不同地址」的散射。所以只有灰度化能向量化，累加不能。

**这也解释了为什么 NEON 提升有限**：直方图统计的真正瓶颈往往是**内存访问 + scatter 累加**，而不是灰度化的算术。灰度化加速了，累加没加速，整体只快一点，甚至在某些环境（如模拟器虚拟化）下反而略慢。这是一个真实且重要的性能分析结论。

**精度损失的根本原因**：NEON 用 `float32`（单精度，尾数 23 位），而方法一/二用 `double`（双精度，尾数 52 位）。

- 权重 `0.299f` 等在 float32 里本身是近似值（0.299 无法被二进制精确表示，double 也不能，但 double 更接近）。
- 逐像素累加时，float32 的中间结果精度更低。
- 当某个像素的真实灰度值恰好落在**半整数附近**（如 `128.5`）时，float32 算出的值可能是 `128.4999` 或 `128.5001`，四舍五入后落到不同的整数 → 该像素被分到**相邻的灰度桶** → 直方图个别桶的计数差 1 → 归一化后表现为「个别 bin ±1」。

所以精度损失是 **float32 精度不足在半整数边界像素上的舍入分歧**，只影响极少数像素（实测网罗 256³ 组合约千分之几），是典型的**性能换精度**权衡。app 主流程默认用**方法二（多线程标量，精确）**，NEON 仅用于对比展示。

### 对比小结

| 方法 | 速度 | 精度 | 用在哪 |
|---|---|---|---|
| 单线程标量 | 最慢（baseline） | 精确 | 对比基准 |
| 多线程标量 | 快 3–4× | 精确 | **app 默认主流程** |
| NEON 多线程 | 最快 | 个别 bin ±1 | 对比展示 |

## 5. 代码逐层讲解

### 5.1 C++ 计算核心 `native/histogram.cpp` + `.h`

对外暴露 3 个 C 函数（`extern "C"`，供 FFI 调用）：

| 函数 | 作用 |
|---|---|
| `hist_compute_rgba(rgba, w, h, out256)` | 默认入口，多线程标量，bin-exact。app 主流程用 |
| `hist_compute_rgba_impl(rgba, w, h, out256, impl)` | 指定实现（`HIST_IMPL_*` = 0/1/2），性能对比用 |
| `hist_compute_gray(gray, w, h, stride, out256)` | 直接统计灰度平面（相机 YUV 的 Y 通道），实时相机用 |

- 返回值都是 `double` = 「统计+归一化」耗时（毫秒）。
- `out256` 是调用方分配的 `int32[256]`，函数写入归一化结果（0–100）。
- `HIST_EXPORT` 宏 = `__attribute__((visibility("default"), used))`，防止 iOS 静态链接时符号被裁掉（详见第 8 节移植注意）。

关键实现细节：
- `gray_of()`：标量灰度化，`std::lround(r*0.299+g*0.587+b*0.114)`。
- `accumulate_scalar()` / `accumulate_neon()`：一个行区间的局部直方图统计。
- `accumulate_parallel()`：分块多线程 + 局部直方图无锁归并。
- 编译加 `-ffp-contract=off`：禁止 FMA（乘加融合）指令，保证浮点结果**跨平台确定一致**（否则 Android/iOS 可能因 FMA 舍入不同而结果有别）。

### 5.2 FFI 绑定层 `lib/ffi/histogram_ffi.dart`

Dart 通过 `dart:ffi` 调用 C 函数：
- `_openLibrary()`：Android `DynamicLibrary.open('libhistogram.so')`；iOS `DynamicLibrary.process()`（符号已静态链入 Runner）。
- `computeHistogram(rgba, w, h)`：分配 native 内存 → 拷贝像素 → 调 C 函数 → 读回 256 个值 + 耗时 → 释放内存。返回 `HistogramResult(normalized, elapsedMs)`。
- `compareImplementations(...)`：跑三种实现，返回 `List<ImplResult>`（含耗时、加速比、精度偏差）。
- `computeGrayHistogram(...)`：灰度平面版，实时相机用。

### 5.3 UI 层 `lib/ui/`

- `home_page.dart`：主页。选图（相册/拍照）、内置渐变图、内置真实样张、三实现对比、实时相机入口。选图后流程：`image_picker` 拿文件 → `dart:ui` 解码成 RGBA → `computeHistogram` → 显示。
- `histogram_painter.dart`：`CustomPainter`，把 256 个归一化值画成 256 根黑色竖柱（白底）。
- `camera_page.dart`：`camera` 插件预览流，每帧取 YUV 的 Y 平面 → `computeGrayHistogram` → 实时更新直方图。用 `_busy` 标志跳过还没算完的帧，避免堆积。

### 5.4 双端接入

- **Android**：`android/app/build.gradle.kts` 的 `externalNativeBuild` 指向 `native/CMakeLists.txt`，NDK 把 `histogram.cpp` 编成 `libhistogram.so` 打包进 APK。
- **iOS**：`native/histogram_core.podspec` 作为 CocoaPods 本地 pod，把 `histogram.cpp` 编入 Runner，符号静态可见。

## 6. 术语与数字词典

| 术语 / 数字 | 含义 |
|---|---|
| **256×100** | 直方图分辨率。256 = 灰度值范围 0–255（横轴）；100 = 归一化后计数的最大值（纵轴） |
| **0.299 / 0.587 / 0.114** | BT.601 灰度化权重（R/G/B），和为 1.0。人眼对绿最敏感故 G 最大 |
| **归一化到 0–100** | 把各灰度级像素计数按 `计数/最大计数×100` 缩放。最多的那级 = 100 |
| **bin / 桶** | 直方图的一个灰度级格子。共 256 个 bin，`bin[i]` = 灰度值 i 的像素数 |
| **bin-exact** | 逐 bin 完全一致，即与标准公式算出的直方图每个桶都相同（零误差） |
| **≤ 300ms** | 核心评分线：直方图生成（统计+归一化）耗时上限 |
| **计时口径** | 只测统计+归一化，不含解码/绘制。保证测的是纯算法性能 |
| **-ffp-contract=off** | 编译选项，禁 FMA 乘加融合，保证浮点结果跨平台确定一致 |
| **Q16 定点** | 曾试的整数近似方案（`(R*19595+G*38470+B*7471+32768)>>16`），因半整数 tie 无法与标准公式对齐而**放弃**，现用 double |
| **NEON** | ARM 的 SIMD 指令集，一条指令处理多个数据 |
| **SIMD** | Single Instruction Multiple Data，单指令多数据（向量化） |
| **scatter（散射）** | 每个元素写到数组不同位置（如 `hist[gray]++`），SIMD 不擅长，是直方图累加的瓶颈 |
| **float32 / double** | 单精度（23 位尾数）/ 双精度（52 位尾数）浮点。NEON 用 float32 故有微小精度损失 |
| **加速比** | 某实现耗时相对单线程 baseline 的倍数，如 4× = 快 4 倍 |
| **局部直方图归并** | 每线程各算一份 256 直方图，最后整数相加。无锁，多线程核心技巧 |
| **YUV / Y 通道** | 相机常用色彩格式。Y = 亮度 = `0.299R+0.587G+0.114B` = **正好是标准灰度公式**，故相机帧可直接统计不用转 RGB |
| **FFI** | Foreign Function Interface，Dart 调用 C/C++ 的机制 |

## 8. 微信小程序移植指南

微信小程序技术栈是 **JS/TS + WXML + WXSS**，**不支持 Flutter、不支持 dart:ffi、不能直接调 C++**。所以移植 = UI 全部重写 + 计算核心换一种方式接入。好消息是：**算法逻辑、公式、数字口径可以 100% 照搬**，直方图算法本身很简单。

### 8.1 各层怎么对应

| 本项目 | 小程序对应 | 说明 |
|---|---|---|
| Flutter UI (`lib/ui/`) | WXML + WXSS + 页面 JS | 全部重写 |
| 直方图绘制 `CustomPainter` | Canvas 2D 画 256 根矩形 | 逻辑照搬 |
| 选图 `image_picker` | `wx.chooseMedia` | |
| 解码取像素 `dart:ui` | Canvas 2D `ctx.getImageData()` | 拿到 RGBA `Uint8ClampedArray` |
| 实时相机 `camera` 插件 | `<camera>` 组件 + `CameraContext.onCameraFrame` | 回调直接给 RGBA `ArrayBuffer` |
| C++ 计算核心 | **见下方两条路** | 二选一 |

### 8.2 计算核心：两条移植路

#### 路 A：纯 JS 重写（推荐先用，最快上手）

直方图算法极简单，JS 重写约 40 行，逻辑与 `histogram.cpp` 一一对应：

```js
// rgba: Uint8ClampedArray (RGBA), 长度 = w*h*4
// 返回 { normalized: number[256], elapsedMs }
function computeHistogram(rgba, w, h) {
  const t0 = Date.now();
  const hist = new Uint32Array(256);
  for (let i = 0; i < w * h; i++) {
    const r = rgba[i*4], g = rgba[i*4+1], b = rgba[i*4+2];
    // 标准公式，四舍五入（与 C++ 一致）
    let gray = Math.round(r * 0.299 + g * 0.587 + b * 0.114);
    if (gray > 255) gray = 255;
    hist[gray]++;
  }
  let max = 0;
  for (let i = 0; i < 256; i++) if (hist[i] > max) max = hist[i];
  const out = new Array(256).fill(0);
  if (max > 0) {
    for (let i = 0; i < 256; i++) {
      out[i] = Math.round(hist[i] * 100 / max);  // 归一化到 0-100
    }
  }
  return { normalized: out, elapsedMs: Date.now() - t0 };
}
```

- **优点**：零工具链，直接写。JS 的 `number` 是 double，精度与 C++ 标量版**一致（bin-exact）**。
- **缺点**：JS 单线程，大图慢。但小程序图片通常不大，且可先缩放到合理尺寸（如长边 ≤ 2000）；实时相机帧一般 ~640×480，单线程也够快。
- **计时**：`Date.now()` 精度较粗（ms 级），可换 `performance.now()` 或 `wx.getPerformance()` 更准。

#### 路 B：Emscripten 把 `histogram.cpp` 编成 WebAssembly（复用 C++，性能好）

小程序支持 `WXWebAssembly.instantiate()` 加载 `.wasm`。用 Emscripten 把**同一份** `histogram.cpp` 编成 wasm：

```bash
emcc native/histogram.cpp -O3 -o histogram.wasm \
  -s EXPORTED_FUNCTIONS='["_hist_compute_rgba","_malloc","_free"]' \
  -s STANDALONE_WASM ...
```

小程序里 `WXWebAssembly.instantiate('/histogram.wasm')` 加载，通过内存指针传 RGBA、调 `hist_compute_rgba`。

- **优点**：复用 C++ 核心，性能好，可保持 double 精确。
- **缺点/限制**：
  - 小程序 wasm **单线程**（无 pthread），所以多线程那套用不了，只能用**单线程标量**（`HIST_IMPL_SCALAR_ST`）。NEON 也用不了（那是 ARM 原生指令，wasm 有自己的 SIMD 但小程序支持有限）。
  - 需要配 Emscripten 工具链，比路 A 复杂。
  - 传像素要手动管理 wasm 内存（malloc/拷贝/free）。

**建议**：先用**路 A（JS）**跑通整个流程，若遇到性能瓶颈再上**路 B（WASM）**。

### 8.3 图像像素怎么拿

**选图路径**：
```js
wx.chooseMedia({ count: 1, mediaType: ['image'], success(res) {
  const path = res.tempFiles[0].tempFilePath;
  // 用 Canvas 2D 画上去再取像素
  const ctx = canvas.getContext('2d');
  const img = canvas.createImage();
  img.onload = () => {
    ctx.drawImage(img, 0, 0, w, h);
    const imageData = ctx.getImageData(0, 0, w, h);  // imageData.data = RGBA
    const result = computeHistogram(imageData.data, w, h);
  };
  img.src = path;
}});
```

**实时相机路径**：
```js
const ctx = wx.createCameraContext();
const listener = ctx.onCameraFrame((frame) => {
  // frame.data 是 ArrayBuffer(RGBA)，frame.width / frame.height
  const rgba = new Uint8ClampedArray(frame.data);
  const result = computeHistogram(rgba, frame.width, frame.height);
  // 更新直方图显示（注意节流，用 busy 标志跳过还没算完的帧）
});
listener.start();
```

> 注意：小程序 `onCameraFrame` 给的是 **RGBA**，不是 YUV，所以要走 `computeHistogram`（含 RGB→灰度），不能用本项目 iOS/Android 那个「Y 通道直接统计」的捷径。逻辑上照 JS `computeHistogram` 即可。

### 8.4 直方图绘制（Canvas 2D）

```js
function drawHistogram(ctx, normalized, cw, ch) {
  ctx.fillStyle = '#fff'; ctx.fillRect(0, 0, cw, ch);        // 白底
  ctx.fillStyle = '#000';
  const colW = cw / 256;
  for (let i = 0; i < 256; i++) {
    const barH = normalized[i] / 100 * ch;                    // 归一化值映射到画布高
    ctx.fillRect(i * colW, ch - barH, colW + 1, barH);        // 从底部向上画柱
  }
}
```

### 8.5 移植工作量与口径一致性

- **必须保持一致**（否则结果对不上本项目）：灰度公式 `0.299/0.587/0.114`、四舍五入、归一化 `计数/max×100`、max==0 特判、256×100 分辨率、计时只算「统计+归一化」。
- **工作量估计**：UI 页面重写（主要工作量）+ 计算核心用路 A 约半天；相机接 `onCameraFrame`。核心算法逻辑本文件已给出完整 JS 版，可直接用。
- **对不上时怎么排查**：先用同一张图，对比本项目和小程序的 256 个归一化值；若个别差 1，检查取整方式（`Math.round` vs 其它）；若差很多，检查像素通道顺序（RGBA vs BGRA）和是否漏了 alpha。
